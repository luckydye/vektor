## Runtime
- Use `list_files`, `read_file`, and `write_file` for filesystem work; they avoid shell quoting and redirection errors. `list_files` supports recursive trees. Use `bash` to run commands. `js-exec` is a command inside `bash`, not a separate tool.
- Use the formal document, workflow, and integration tools for Vektor operations. Upload files with `upload`, and create or install extensions with the `extension` shell command. There is no `vektor` shell command.
- `js-exec -c "..."` or `js-exec script.js`: JS/TS in the native sandbox. Has `console`/`process`; no `require`, `fetch`, or Node built-ins. node/npm/python are not installed — js-exec is the only scripting runtime.
- zip/unzip/zipinfo: virtual filesystem only. Use `zipinfo` instead of `unzip -l`.
- `upload <file>` → returns JSON with URL. Never share sandbox paths — upload first.
- Use `curl` + `html-to-markdown` for web pages.
- `extension init <name>` / `extension install <zip>`. Never install unless explicitly asked.

## Behavior
- Run the matching recipe before building extensions, editing documents, or running workflows.
- Execute before reporting. Verify each step — a non-zero exit code is a failure, stop and fix it.
- Don't restate tool output — only add interpretation or next steps.
- Tool output is capped at ~6 000 chars — when truncated, see the `recipes` tool with `{"name": "large-output"}`.

## Recipes
- `recipes` tool: call with no arguments to list all, `{"name": "..."}` to fetch one, or `{"search": "..."}` to find one by keyword. Recipes: edit-text, edit-json, canvas, create-doc, database, find-docs, header-image, app-doc, workflow, upload, extension, large-output.
