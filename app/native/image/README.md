# native/image

A small Rust [N-API](https://nodejs.org/api/n-api.html) addon that provides the
subset of image processing the app needs — **resize + format conversion +
metadata** — as a replacement for `sharp`.

## Why

`sharp` ships a native libvips binary (or a WASM build) that Bun cannot cleanly
embed into a single-file executable. A plain `.node` N-API addon, by contrast,
is [embedded automatically](https://bun.com/docs/bundler/executables#embed-n-api-addons)
by `bun build --compile` when it's referenced by a static `require()`.

## Build

```sh
bun i          # once, to install @napi-rs/cli
bun run build  # → ../../src/files/native/image-<platform>-<arch>.node
```

`task setup` (repo root) runs this automatically.

The output filename encodes Bun's `process.platform`/`process.arch` so a
checkout can hold addons for multiple platforms side by side. The artifacts are
git-ignored — they're rebuilt by `task setup` locally and per-runner in CI.

## How it's wired in

- `src/files/native.ts` loads the addon. In the compiled binary, `build.ts`
  injects the embedded path via `--define VEKTOR_NATIVE_ADDON=...` so Bun traces
  and embeds the `.node`. In dev/tests it falls back to the host addon on disk.
- `src/files/transforms.ts` calls `transform()` / `metadata()`.
- The test suite uses `encodeSolid()` to mint fixtures.

## Cross-platform releases

A native addon is architecture-specific, so the release builds on **native
runners** (matrix in `.github/workflows/release.yml`): each runner builds its
own `.node` and compiles its own target binary — no cross-compilation.

## API

See `generated/index.d.ts` (auto-generated by NAPI-RS):

- `transform(input, { w, h, format, quality })` — decode → optional
  fit-inside resize (never enlarges) → encode. `format: ""` keeps the input
  format. WebP/JPEG honour `quality`; PNG/GIF are lossless.
- `metadata(input)` — `{ width, height, format }`.
- `encodeSolid(width, height, r, g, b, format, quality)` — a solid-colour image
  (test fixtures).
